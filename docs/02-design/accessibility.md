# 접근성 (Accessibility)

## 의사결정 근거

- 학습 제품은 장시간 사용되므로 가독성/키보드 탐색/포커스 처리 품질이 곧 제품 품질
- 접근성은 장애를 가진 사용자만을 위한 것이 아니라, 모든 사용자의 경험을 향상시킴

---

## 기본 품질 요구사항

### 1. 색상 대비

- **목표**: APCA (WCAG 3.0 후보) Silver/Gold 수준
- **본문 텍스트**: Lc 75 이상 권장 (최소 Lc 60)
- **큰 텍스트 / UI**: Lc 60 (18px+) ~ Lc 45 (24px+ Bold)
- **특이사항**: 폰트가 얇을수록 더 높은 대비 필요, 배경 밝기에 따라 인지 특성 반영

### 2. 포커스 표시

- 모든 인터랙션 요소에 명확한 포커스 링
- 포커스 상태가 배경과 충분히 구분되어야 함

### 3. 키보드 탐색

- 키보드만으로 모든 기능 사용 가능
- 키보드만으로 세션 진행 가능
- 논리적인 탭 순서

---

## 구체적 가이드라인

### 색상 및 시각 요소

- **포커스 표시**: 현재 어떤 요소를 선택하고 있는지 명확히 알 수 있도록 강조 테두리(포커스 링)를 제공합니다. 강조색을 사용하여 배경과 뚜렷하게 구분합니다.
- **가독성 확보**: 텍스트와 배경의 대비를 높여 눈의 피로를 예방하고 가독성을 확보합니다.

### 키보드 탐색 및 조작

사용자가 마우스 없이 키보드만으로도 모든 기능을 막힘없이 사용할 수 있도록 지원합니다.

| 키                | 동작 설명                          |
| ----------------- | ---------------------------------- |
| `Tab`             | 다음 버튼이나 입력창으로 이동      |
| `Shift + Tab`     | 이전 버튼이나 입력창으로 뒤로 이동 |
| `Enter` / `Space` | 현재 선택된 버튼 누르기            |
| `Escape`          | 열려 있는 창이나 팝업 닫기         |
| `방향키`          | 목록이나 메뉴 항목 사이를 이동     |

### 보조 공학 도구 지원 (스크린 리더)

시각 정보가 부족한 사용자를 위해 화면 낭독기가 제품의 상태를 정확하게 전달할 수 있도록 합니다.

- **버튼 및 링크**: 아이콘으로만 이루어진 버튼도 그 기능을 설명하는 텍스트 정보를 내부에 포함하여 무엇을 하는 버튼인지 읽어주도록 합니다.
- **학습 진행률**: 현재 몇 % 정도 학습이 완료되었는지 정보를 텍스트로 함께 제공하여 진행 상황을 인지할 수 있게 합니다.
- **구조화된 정보**: 탭 메뉴나 리스트처럼 연관된 항목들은 그 그룹 관계와 현재 선택 상태를 명확히 알 수 있도록 정보를 구성합니다.

### 모션 최적화

OS 설정에서 애니메이션을 줄이도록 설정한 경우, 시각적인 흔들림이나 화려한 효과를 최소화하여 원활한 학습 환경을 제공합니다.

## 테스트 체크리스트

### 수동 테스트

- [ ] 키보드만으로 모든 페이지 탐색 가능
- [ ] 포커스 상태가 명확히 보임
- [ ] 색상 대비 충분함
- [ ] 스크린 리더로 콘텐츠 이해 가능
- [ ] **학습 세션 전체를 키보드로만 완료 가능**
- [ ] **화면낭독기(NVDA, JAWS, VoiceOver)로 주요 흐름 테스트**

### 자동 테스트

- [ ] axe-core 또는 Lighthouse 접근성 검사 통과
- [ ] ESLint a11y 규칙 준수
- [ ] Wave 또는 동등 도구로 검증

### 실무 팁

- 2025년부터 접근성 준수가 법적 의무화되는 추세
- 프로젝트에서 접근성을 고려한 것은 큰 강점

## 가이드라인

### DO ✅

- 명확한 포커스 표시
- 충분한 색상 대비
- 키보드 접근성 보장
- 스크린 리더 지원

### DON'T ❌

- 마우스 전용 인터랙션
- 색상만으로 정보 전달
- 포커스 링 제거
- 자동 재생 콘텐츠
